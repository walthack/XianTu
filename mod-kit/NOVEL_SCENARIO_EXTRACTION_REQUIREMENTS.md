# 小说剧情 Mod 提取与生成要求

本文档用于指导 Agent 从小说正文、章节概要、人物卡和设定资料中提取阶段剧情，并生成可导入 XianTu 的 Scenario Mod。目标不是复述整部小说，而是制作一个时间点明确、人物状态正确、能够从开场持续游玩的剧情剧本。

生成前必须同时阅读：

- `AGENT_GUIDE.md`
- `schema/xiantu.scenario-mod.v1.schema.json`
- `templates/strict.template.json`
- 本文档

本文档的人工交付要求可能严于当前 JSON Schema。Schema 未自动拦截的要求仍然是交付门槛，必须由来源记录、关系审计和二次审核确认，不能因为 validator 未报错而省略。

## 一、交付目标

每个 Mod 只覆盖一个明确剧情阶段，类似“黄巾之乱”“三国鼎立”，必须满足：

1. 玩家进入时的时间点明确。
2. 世界、势力、人物、地点、能力和装备符合该时间点。
3. 开场事件可以立即游玩。
4. 章节和事件能够按 flag 继续推进。
5. 正典人物不会被 AI 随意改名、换宗派或获得他人专属能力。
6. 存档重载后仍保留 Mod 身份、当前章节和剧情 flag。

不要把整本小说逐章塞进一个包。剧情跨度过大时，应拆成多个阶段包。

## 二、素材与证据

### 2.1 素材优先级

按以下顺序确定事实：

1. 小说正文。
2. 与正文对应的章节概要。
3. 官方或项目内人物卡、设定表。
4. 背景分析和二手摘要。

低优先级材料与正文冲突时，以正文为准。无法确认的事实不得自行补成正典。

### 2.2 提取记录

交付时必须附一份 `<mod-name>.sources.md`，至少记录：

| 字段 | 内容 |
|---|---|
| 剧情阶段 | 起止章节或明确时间点 |
| 主要素材 | 文件名、章节范围 |
| 开场依据 | 对应章节和事件 |
| 人物依据 | 每名正典人物首次/当前状态来源 |
| 关系依据 | 主角关系、NPC 关系对应依据 |
| 能力依据 | 技能、功法、装备获得时间 |
| 不确定项 | 未采用或降级为补充设定的内容 |

禁止复制大段小说正文。只写事实摘要和必要的短句证据。

## 三、阶段边界

### 3.1 必须确定的边界

生成 JSON 前先列出：

- 开场发生在哪一章、哪一天或哪个事件之后。
- 本阶段已经发生的关键事件。
- 本阶段尚未发生、不得提前泄露的事件。
- 已经加入或离开的角色。
- 已死亡、失踪、被囚或暂时不在场的角色。
- 主角此时已经掌握和尚未掌握的能力、功法、装备。

### 3.2 禁止跨阶段污染

- 后期人物不得提前出现在早期阶段。
- 已死亡人物不得作为普通可互动 NPC 初始化。
- 尚未获得的技能、功法、装备不得开局发放。
- 后续关系不得提前写成当前好感或盟友关系。
- 后续真相只能作为未激活章节内容，不得放入开场提示。

历史人物若仍影响剧情，应写入人物记忆、事件背景或章节摘要；除非存在灵魂、回忆交互等明确机制，否则不要加入 `canon.playerRelationships`。

当前 Schema 没有人物 `isAlive` 字段。已死亡且不可互动的人物原则上不要放入 `canon.characters`；只在事件摘要、章节背景或其他人物记忆中引用，并在关系审计中明确记录其死亡状态。

## 四、玩家路线与创角预制

### 4.1 正典主角路线

玩家扮演小说主角时必须设置：

```json
"playerCharacterId": "mod.character.protagonist"
```

并提供 `scenario.opening.creationPreset`，完整定义创角第 2 至第 7 步：

- `talentTier`：该阶段的角色起点，不套用仙途通用等级。
- `origin`：原作出身和当前身份。
- `spiritRoot`：原作灵根、血脉、体质或特殊能力；原作没有传统灵根时必须明确说明。
- `talents`：角色本阶段已经具备的稳定特质。
- `attributes`：根骨、灵性、悟性、气运、魅力、心性，范围 0–10。
- `characterName`、`gender`、`race`、`age`。
- `locked: true`：正典身份默认锁定。

设置 `playerCharacterId` 的交付包必须同时提供 `creationPreset`。当前 validator 尚未强制这两个字段成对出现，审核者必须人工检查。

预制内容必须随剧情阶段变化。不能把后期成熟状态复制到早期包。

### 4.2 独立玩家路线

不填写 `playerCharacterId` 表示原创玩家。此时不得让玩家自动继承任何正典 NPC 的宗派、技能、功法或装备。

同一个 JSON 当前只应提供一种明确玩家身份。需要“正典主角”和“原创角色”两种玩法时，分别交付两个 Mod 文件，避免关系和专属内容语义冲突。

## 五、人物清单

### 5.1 收录标准

仅收录本阶段满足至少一项条件的人物：

- 开场直接出现。
- 是章节目标或关键事件参与者。
- 与玩家有持续关系。
- 与其他关键 NPC 有影响剧情的关系。
- 掌握本阶段必须保护的正典内容。

一次性路人没有姓名依据时，不得伪造为原作人物。可用明确标注的功能性角色，但必须在来源文档中说明是“可玩性补充”。

### 5.2 人物状态

每名人物必须核对：

- 姓名和别名。
- 性别、角色定位和当前境界。
- 当前地点。
- 宗派、军队、国家、家族、组织归属。
- 该阶段是否存活并可互动。
- 当前已掌握技能、功法和持有物品。

宗派等归属必须写入 `affiliations`，不能只埋在描述中。

### 5.3 当前原生映射限制

当前 `canon.characters` 可以准确提供姓名、性别、定位、境界、地点、归属和内容引用，但原生 NPC 初始化仍有以下限制：

- NPC 种族暂时按人族初始化。
- NPC 出生日期暂用系统默认值。
- NPC 先天六司暂用统一基础值。
- 未提供 `realm` 时会回退为凡人。

Agent 不得在 JSON 中添加 Schema 不支持的字段规避限制。若种族、年龄或六司是剧情判定的关键条件，应在最终报告中标为“模块能力阻塞”，不得声称已经完成原生映射。

## 六、关系与好感

### 6.1 玩家关系

所有需要在游戏开始时出现在原生人物关系列表中的 NPC，都必须写入：

```json
"canon.playerRelationships"
```

每条必须包含：

- `characterId`
- `relation`
- `favorability`，范围 -100 至 100
- 有明确共同经历时填写 `memories`

不得仅把人物写入 `events[].relatedCharacterIds`。事件参与不会自动创建原生 NPC。

注意：出现在 `canon.relationships` 的 NPC 端点也会被当前运行时初始化，并在缺少玩家关系时默认显示为“陌生人 / 好感 0”。为避免玩家提前认识不应出现的人物，所有开场可见人物应明确写入 `playerRelationships`；不应出现在玩家列表的人物不要仅为了保存关系而过早加入关系边。

### 6.2 NPC 之间的关系

只有 NPC 与 NPC 的关系写入：

```json
"canon.relationships"
```

玩家映射为正典人物后，所有涉及 `playerCharacterId` 的关系边会被运行时过滤，不进入 NPC 关系矩阵。因此主角与其他人物的关系必须同时、且以 `playerRelationships` 为权威定义。

玩家关系强度字段叫 `favorability`；NPC-NPC 关系强度字段叫 `score`。两者范围均为 -100 至 100，不得混用字段名。

### 6.3 关系一致性

- 同一对人物的关系名称和强度必须一致。
- `师徒` 与 `授业者`、`护卫与伙伴` 与 `护卫与主事者` 等近义描述不得在两套关系中混用。
- 好感值必须反映开场时间点，不能使用最终关系。
- 敌对、戒备、陌生、利用、复杂盟友等关系不能全部粗略写成“朋友”。

### 6.4 覆盖检查

交付前生成以下清单：

| 人物 | 是否原生初始化 | 与玩家关系 | 好感 | 记忆 | NPC 关系 |
|---|---:|---|---:|---|---|

任何 `canon.characters` 若既不在 `playerRelationships`、也不在 `relationships`，必须说明为什么无需初始化；否则视为遗漏。

## 七、势力、地点与正典内容

### 7.1 势力与地点

- 每个势力使用 Mod 内自定义且唯一的 ID。
- 人物所属势力必须引用已有 `factionId`。
- 开场地点必须存在于 `canon.locations`。
- 地点所属大陆与势力必须符合阶段事实。
- 已声明内容不得被 AI 同名替代。

### 7.2 技能、功法和装备

- `content.skills`：能力、招式或特殊异能。
- `content.techniques`：功法、传承和修炼体系。
- `content.items`：武器、护甲、消耗品、材料和剧情物品。
- 人物通过 `skillIds`、`techniqueIds`、`itemIds` 关联内容。
- 对被初始化的 NPC 或由 `playerCharacterId` 映射的正典玩家，`skillIds`、`techniqueIds`、`itemIds` 会映射为开局技能、功法物品和物品；因此这些引用必须严格符合获得时间。
- 多条 `techniqueIds` 都会进入背包，但当前只有数组第一条作为开局主修功法。必须按主修优先级排序。

唯一能力使用 `rules.contentAccess.policy = "exclusive"`；限定传承使用 `restricted`。例如生死根只能属于程宗扬，其他 NPC 和原创玩家不能通过 AI 指令获得。

## 八、剧情章节与事件

### 8.1 开场

`scenario.opening.text` 必须说明：

- 玩家是谁。
- 玩家在哪里。
- 当前正在发生什么。
- 玩家立即面临的选择或危险。

不要把整个阶段摘要写成开场，也不要泄露后续章节。

### 8.2 章节

每章必须具备：

- 明确目标。
- 可满足的激活条件。
- 可满足的完成条件。
- 至少一个与目标相关的事件，终章除外。

非终章不得没有完成条件。章节之间不得形成依赖环。

### 8.3 事件与 flag

- 所有条件使用的 flag 必须在 `scenario.initialFlags` 初始化。
- 事件完成事实必须能从玩家行为或叙事中判定。
- 不使用“剧情应该已经发生”这类不可验证条件。
- 未激活章节和未来事件不得进入当前故事提示。

## 九、禁止事项

以下任一项出现即退回：

- 把摘要中的推测写成原作事实。
- 混入另一部小说、同作者其他作品或同名人物。
- 使用未核对的占位姓名冒充正典角色。
- 已死亡人物作为普通 NPC 初始化。
- 正典主角同时作为玩家和 NPC 出现。
- 原创玩家继承正典人物专属能力。
- 把所有人物默认设置为凡人、16 岁、五行灵根或统一六司。
- 只定义人物，不定义其关系，导致人物无法进入原生系统。
- 使用 NPC 关系边代替玩家关系。
- 生成整本小说式超大包。

## 十、交付文件

每个阶段至少交付：

1. `<mod-name>.json`
2. `<mod-name>.sources.md`
3. `<mod-name>.relationship-audit.md`
4. 校验输出

关系审计文件必须列出人物覆盖表、存亡状态、玩家关系和被刻意排除的人物。

## 十一、自动校验

在项目根目录执行：

```bash
npm run mod:validate -- path/to/mod.json
npm test
npm run type-check
```

`npm run mod:validate` 会先运行结构 validator，再运行静态剧情 analyzer；其输出同时包含引用/字段错误以及 flag、章节可达性、阻塞章节、循环依赖和孤立事件诊断。

正式交付要求：

- Validator：0 error。
- 静态剧情分析：0 error，warning 必须逐条解释。
- JSON Schema：通过。
- 不得修改现有测试以掩盖失败。

以下要求目前主要依靠人工审核，而非 Schema 强制：存亡状态、`playerCharacterId` 与 `creationPreset` 成对、单包单玩家身份、人物关系覆盖，以及 NPC 种族/年龄/六司映射限制。

## 十二、实机验收

每个阶段包都必须单独完成，不得只测试第一包：

1. 首页进入“剧本模组”。
2. 导入 JSON 并检查预览。
3. 选择替换旧版本。
4. 进入创角并选择 Mod。
5. 检查第 2–7 步是否使用正确预制或自由创角规则。
6. Strict 模式确认不调用 AI 世界生成。
7. 生成开场并进入游戏。
8. 检查玩家姓名、出身、特殊能力、技能、功法和装备。
9. 检查原生人物关系列表、关系名称、好感和记忆。
10. 确认玩家本人未重复成为 NPC。
11. 保存、返回主页、重新加载。
12. 确认 Mod ID、当前章节、flag、正典内容和关系仍然存在。

## 十三、Agent 最终报告格式

Agent 完成后必须按以下格式报告：

```text
Mod ID：
阶段范围：
玩家身份：正典人物 / 独立玩家
开场地点：
人物数：
玩家关系数：
NPC 关系数：
势力 / 地点 / 技能 / 功法 / 物品数：
章节 / 事件数：
历史或死亡人物处理：
不确定项：
Validator：PASS / FAIL
静态分析：PASS / FAIL
实机闭环：PASS / FAIL
```

未完成来源记录、关系审计或实机闭环时，不得声明 Mod 可交付。
